# GHB-PPT-SKILL 工作流与模型能力依赖分析报告

> 分析日期：2026-07-13  
> 分析对象：当前仓库中的 `SKILL.md`、统一入口 `scripts/ghb_ppt.py`、作者工作流、SVG/版式/质量规则、构建脚本、验证脚本与示例项目。  
> 核心问题：整个工作流中，哪些环节高度依赖模型能力，哪些环节主要依赖确定性程序能力。

## 1. 结论摘要

当前 SKILL 是一条“模型负责内容与视觉决策，程序负责工程化生产与验证”的混合流水线，不是端到端由模型直接生成 PPTX。

- **模型能力依赖最大**的环节集中在构建前：理解源材料、确认需求、内容取舍、逐页叙事、结论式标题、内容到版式的映射、SVG 信息设计、视觉复核与非确定性修复。
- **模型能力依赖较小**的环节集中在构建后：环境检查、项目初始化、模板解析、封面填充、字体修复、SVG 规则检查、白底移除、SVG 后处理、DrawingML 导出、OOXML 合并、结构验证、渲染和报告生成。
- **真正决定成品上限的是高模型依赖区**。后半段已经能较好保证“能构建、可编辑、结构正确、没有明显技术错误”，但不能自动保证“内容有洞察、叙事有节奏、版式语义匹配、整体审美优秀”。
- **真正决定稳定下限的是低模型依赖区**。统一 CLI、质量门、OOXML 合并、PPT 回读、渲染及检查点机制把大量易出错工作从模型自由发挥中移出，使结果更可复现。
- 以正常的“GHB 模板 + 结构化正文”项目估算，若按工作流决策重要性而非运行耗时计，约 **55%–65% 的质量责任依赖模型/人工判断，35%–45% 依赖确定性工具链**。若只按机器执行步骤数量计，则程序步骤更多；若按最终视觉与内容质量贡献计，则模型侧更重。

### 2026-07-13 修复实施结果

本报告提出的可确定性修复已落地：

| 原问题 | 修复状态 | 实施结果 |
|---|---|---|
| 六项确认只存在于文档，未强制执行 | 已修复 | 新增 `confirmation.json`、决策摘要和 `check-project`；`check-svg`、`build-content`、`merge`、`build` 均强制执行，完整构建在写封面前 fail-fast |
| fixture 可冒充真实用户确认 | 已修复 | `fixture` 仅允许仓库示例/测试目录或测试运行器注入的 `GHB_PPT_TEST_FIXTURE=1` 环境；正常 CLI 无绕过参数 |
| 缺少“内容模型 → 页面模型”中间层 | 已修复 | 新增必需 `content_model.json`，不可删减论点必须有来源并通过 `claim_ids` 映射到页面 |
| 版式多样但语义可能错误 | 已修复首批硬规则 | timeline、matrix、swimlane、flywheel、comparison 分别强制顺序、双轴、owner、闭环和共享标准证据 |
| `layout_plan.json` 只有局部检查 | 已修复 | 强制逐页 schema，不再静默丢弃畸形行；确认大纲、节奏、受众和 SVG 数量必须与实际计划一致 |
| 确认后可静默修改决策 | 已修复 | `decision_digest` 绑定六项决策；改动后必须重新确认并更新时间戳/摘要 |
| 自由 SVG 的视觉上限依赖模型 | 结构性缓解 | 保留组件 schema、Office-safe 质量门和渲染复核；审美与信息设计仍不能伪装成完全确定性问题 |
| 字体环境削弱视觉反馈 | 环境限制保留 | 继续由 `doctor` 明确报告；缺少 Microsoft YaHei 时仍禁止宣称中文视觉保真通过 |

## 2. 评分口径

本文用“模型能力依赖度”衡量某环节是否必须依靠大模型的语义理解、推理、创作或视觉判断，而不是衡量该环节耗时。

| 等级 | 模型能力占比参考 | 含义 |
|---|---:|---|
| 极高 | 80%–100% | 缺少强模型或人工专家时，难以得到合格结果；规则只能约束，不能替代决策 |
| 高 | 60%–79% | 模型主导，程序可提供模板、候选项和硬约束 |
| 中 | 40%–59% | 模型判断与确定性工具同等重要 |
| 低 | 15%–39% | 程序主导，模型主要负责解释异常或选择参数 |
| 极低 | 0%–14% | 可以稳定地由脚本完成，模型只负责调用和汇报 |

占比是架构分析值，不是通过性能实验测得的精确统计值。它表达“该环节产出质量对模型能力的敏感程度”。

## 3. 当前完整工作流

```text
请求进入
  → 路径判断
  → doctor / init / analyze-template
  → 读取和归一化源内容
  → 六项内容确认闸门
  → design_spec / spec_lock
  → 逐页大纲与 layout_plan
  → 可选图片、图标与版权策略
  → 正文 SVG 创作 + speaker notes
  → authored SVG 质量门
  → 封面填充与字体修复
  → 白底移除 / SVG finalized / 可编辑 DrawingML 导出
  → 封面、正文、母版、主题与致谢页 OOXML 合并
  → PPTX 回读与结构验证
  → PDF / PNG / contact sheet 渲染
  → 逐页视觉复核
  → 确定性修复或内容/视觉返工
  → 最终报告、日志、状态文件与全部产物交付
```

统一 `build` 命令实际覆盖的是封面、正文构建、合并、预渲染验证、可用时渲染、最终验证和报告。它**不负责自动理解源内容、自动完成需求确认、自动制定高质量逐页叙事，也不提供通用的内容到 SVG 生成器**。这些仍是模型/作者侧职责。

## 4. 分环节模型能力依赖矩阵

| 阶段 | 主要产物/动作 | 模型依赖度 | 参考占比 | 判断依据 |
|---|---|---:|---:|---|
| 1. 路径判断 | template-fill、完整 SVG/OOXML 流程或自由演示流程三选一 | 中 | 45% | 规则清晰，但要理解“模板保真、正文结构化表达、仅替换文字”等真实意图 |
| 2. 环境检查 | Python、依赖、模板、字体、渲染器、权限检查 | 极低 | 5% | `doctor` 可确定性完成；模型只需解释 warning/error |
| 3. 项目初始化 | 建立规范目录 | 极低 | 2% | `init` 完全由脚本执行 |
| 4. 模板分析 | 提取页面、槽位、版式信息 | 低 | 20% | 分析脚本可提取结构；模型仍需理解哪些版式适合当前内容 |
| 5. 源内容理解 | 提炼主题、事实、逻辑、受众价值 | 极高 | 90% | 信息压缩、隐含关系识别和重点判断无法由结构检查器替代 |
| 6. 六项确认闸门 | 受众、页数、模式、大纲、取舍、素材选择 | 极高 | 85% | 需要针对源材料提出选项、推荐并预判取舍影响 |
| 7. 内容规格锁定 | `design_spec.md`、`spec_lock.md` | 高 | 70% | 品牌常量可模板化，但语气、密度、视觉策略和内容边界需判断 |
| 8. 逐页叙事规划 | 页面顺序、节奏、结论式标题、展开/合并 | 极高 | 95% | 是演示文稿的信息架构核心，直接决定可理解性和说服力 |
| 9. 内容到版式映射 | `layout_plan.json`、结构选择、备选方案 | 极高 | 85% | 版式目录只提供候选；选对结构依赖语义建模，而非关键词机械匹配 |
| 10. 素材策略 | 无图、提取、自备、联网或 AI 生图；图标与版权 | 高 | 65% | 文件处理与来源记录可自动化，但“图是否必要、表达什么”依赖判断 |
| 11. 正文 SVG 创作 | 可编辑文本/图形、信息层级、几何布局 | 极高 | 90% | 当前正常工作流没有通用内容到 SVG 生成器；示例生成器不是通用方案 |
| 12. 演讲备注编写 | `notes/total.md` 与逐页备注 | 高 | 75% | 拆分可自动化，备注内容需理解页面目的和讲述逻辑 |
| 13. Authored SVG 检查 | SVG 安全、布局多样性、素材、碰撞、密度 | 极低 | 10% | 三类 checker 基于明确规则执行；模型负责理解告警是否合理 |
| 14. 封面构建 | 模板填充、输出定位、字体规范化 | 极低 | 8% | `build-cover`、template-fill 和 `fix_cover_font.py` 为确定性操作 |
| 15. SVG 正式化 | 白底移除、图标/图片嵌入、位置处理 | 极低 | 8% | `remove_svg_background.py`、`finalize_svg.py` 负责处理 |
| 16. DrawingML 导出 | SVG 转原生可编辑文本和形状 | 极低 | 5% | 转换由程序完成，失败即返回明确错误 |
| 17. OOXML 合并 | 封面、正文、母版、版式、主题、媒体、致谢页 | 极低 | 5% | `merge_template_master.py` 是确定性包级变换 |
| 18. 结构与内容验证 | ZIP、关系、母版、页数、备注、边界、回读等 | 极低 | 10% | `ppt_to_md.py` 与 `validate_ghb_pptx.py` 提供硬门和报告 |
| 19. 渲染 | PDF、逐页 PNG、contact sheet | 极低 | 3% | 有渲染器时完全自动；字体/渲染器缺失属于环境问题 |
| 20. 视觉复核 | 遮挡、乱码、层级、密度、审美、重复几何 | 高 | 75% | 技术异常可部分检测，但审美、语义匹配和整体节奏仍需视觉判断 |
| 21. 确定性修复 | 封面字体、已知 OOXML 合并错误 | 极低 | 10% | `--repair-attempts 0..3` 仅重试已知、可证明安全的修复 |
| 22. 内容/视觉返工 | 重写、换布局、拆页、重做几何 | 极高 | 90% | SKILL 明确禁止自动掩盖未知失败；真正返工重新进入创作决策区 |
| 23. 状态恢复与交付 | checkpoint、日志、报告、产物清单 | 低 | 15% | 状态与文件由程序记录；模型负责汇总限制、避免过度宣称 |

## 5. 高模型依赖区分析

### 5.1 源内容理解与内容取舍

这是整条链路的第一个能力瓶颈。模型必须区分事实、观点、证据、背景和结论，并根据受众决定保留什么、删减什么、解释什么。程序可以统计字数、提取标题，却无法仅靠规则判断“对这个受众最重要的三件事”。

弱模型在这里常见的结果是：照抄原文、所有内容同等重要、结论泛化、页面过密或为了页数随意删减。

### 5.2 六项确认闸门与逐页叙事

确认闸门不是简单收集参数，而是要求模型结合源内容提出具体选项、给出推荐并解释修改影响。逐页大纲还需要控制 `anchor`、`dense`、`breathing` 节奏，形成信息递进。

这是最终 PPT 是否“像一场完整表达”而不是“若干页面拼接”的决定性环节，模型能力占比最高。

### 5.3 内容到版式的语义映射

仓库已提供十种内置结构和更多语义模式，也能检查连续重复及版式数量，但 checker 只能判断“是否多样”，不能判断“是否选对”。例如：

- 时间顺序适合 `timeline`，能力成熟度更适合 `staircase`；
- 因果问题不应机械地做成三个平级卡片；
- 多角色交接只有在确有责任边界时才适合 `swimlane`；
- 反馈循环必须真的构成闭环，才适合 `flywheel`。

从内容关系推导视觉结构，本质上是语义推理任务，因此即使有组件库，也仍高度依赖模型。

### 5.4 SVG 信息设计

Office-safe 规则、颜色、字号下限、边界和碰撞元数据可以被程序约束，但信息层级、留白、视觉重心、对齐节奏、图形隐喻和局部几何仍需要创作能力。

当前示例项目能用脚本生成特定主题的 SVG，但仓库审计已经明确：正常工作流里的 SVG 由 agent 创作，示例生成器并不是通用内容到 SVG 引擎。因此该环节仍是最大的模型能力和稳定性风险之一。

### 5.5 视觉复核与非确定性返工

结构验证能发现越界、关系错误、整页图片、文本缺失等问题，却不能可靠回答：

- 页面是否一眼可读；
- 重点是否足够突出；
- 图形是否真正解释内容；
- 连续页面是否有节奏；
- 某种版式虽合规但是否显得机械、单调或失衡。

因此最终 contact sheet 的逐页复核仍是高模型/人工依赖环节。遇到内容过密、几何薄弱或结构不匹配时，必须重新进入“重写 → 换布局 → 拆页 → 调间距 → 小幅调字号”的创作循环。

## 6. 低模型依赖区分析

### 6.1 环境、目录和模板结构处理

`doctor`、`init`、`analyze-template` 已把依赖、字体、渲染器、权限、模板槽位等工作变成可复现的程序输出。模型不应猜测环境状态，只需读取结果并决定是否继续或记录限制。

### 6.2 SVG 技术质量门

Authored 和 finalized 两道门能够检查 Office-safe SVG、版式多样性、图片和图标规则、碰撞元数据、乱码、文本负载、白底等。这类“明确规则 + 明确失败条件”最适合程序化，模型不应替代 checker 目测猜测。

### 6.3 可编辑 PPTX 构建

白底移除、SVG 后处理、DrawingML 转换、母版/主题/媒体注入都是精确的文件格式变换。此处依赖的是脚本正确性、OOXML 知识和测试覆盖，而不是模型临场能力。

将这些步骤固定在统一 CLI 中，显著降低了模型遗漏命令、手工改 ZIP 或使用整页截图冒充可编辑内容的风险。

### 6.4 最终结构验证与渲染

PPT 回读、页数、角色、关系、Content Types、母版链、备注、对象边界、整页图片、计划文字覆盖等均已有程序检查。渲染器可用时，PDF、PNG 和 contact sheet 也能自动产出。

这里的模型责任主要是：不忽略警告、不把结构通过误报为视觉通过，并准确披露 Microsoft YaHei 缺失或未渲染等限制。

### 6.5 已知错误修复与状态恢复

流水线只自动修复封面字体和已知 OOXML 合并错误，且限制为 0–3 次；未知错误立即失败并保留 run 现场。检查点和 `.ghb/state.json` 也使中断恢复成为程序状态问题，而不是依赖模型记忆。

这是合理边界：可证明安全的修复交给程序，涉及内容和审美的修复交还模型或人工。

## 7. 模型能力并非单一维度

高依赖环节实际依赖四类不同能力：

| 能力维度 | 主要影响环节 | 对成品的影响 |
|---|---|---|
| 长文本理解与压缩 | 源内容理解、取舍、备注 | 决定信息是否准确、完整且不过载 |
| 逻辑与叙事推理 | 大纲、页面顺序、结论式标题 | 决定整套 PPT 是否有主线和说服力 |
| 视觉空间与版式推理 | layout plan、SVG、视觉返工 | 决定结构是否匹配语义、页面是否清晰美观 |
| 自我检查与证据意识 | 视觉复核、限制披露、交付 | 决定模型是否会把“构建成功”错误宣称为“质量通过” |

因此，单纯提升语言生成能力不能完全解决当前上限问题；视觉空间推理、可靠的渲染反馈和基于证据的自我纠错同样重要。

## 8. 现有工作流对模型能力的约束效果

当前设计已经通过以下机制降低了对模型“稳定发挥”的依赖：

1. **先确认再创作**：避免模型在受众、页数和叙事模式未明确时直接生成。
2. **先计划再画 SVG**：`layout_plan.json` 强制模型解释版式选择，而不是边写边猜。
3. **设计令牌固定**：画布、品牌色、字体、标准内容区和可编辑性要求被锁定。
4. **版式目录与多样性门**：减少模型反复使用卡片网格的倾向。
5. **两道 SVG 门**：将浏览器可用但 Office/WPS 不安全的实现提前拦截。
6. **统一构建 CLI**：把复杂的转换和 OOXML 操作从自然语言执行中移出。
7. **回读 + 结构 + 渲染证据**：避免仅凭“文件已生成”宣告成功。
8. **有限修复与保留现场**：阻止模型用不透明的反复尝试掩盖根因。

这些机制主要提升的是稳定下限，而不是自动提升内容洞察与视觉创意上限。

## 9. 当前主要能力缺口

### 9.1 缺少通用的“内容模型 → 页面模型”中间层

`layout_plan.json` 已经是很好的起点，但目前仍主要由模型自由编写。缺少对“论点、证据、关系、优先级、受众疑问”的结构化表示，导致内容理解和版式选择之间仍存在一次较大的自由跳跃。

### 9.2 版式选择有规则，但语义适配缺少机器评分

当前可以检查多样性，却不能检查 `timeline` 是否真的包含时间、`matrix` 是否真的存在两条轴、`swimlane` 是否真的存在多个责任主体。这使“形式多样但语义错误”仍可能通过硬门。

### 9.3 SVG 组件库降低了绘图成本，但没有消除组合判断

十种结构型 renderer 能稳定输出 Office-safe 几何，但模型仍要决定内容分组、每组文字长度、强调关系和页面重心。若直接把任意内容塞入组件，仍会出现结构正确但表达薄弱的问题。

### 9.4 视觉检查仍以硬错误为主

当前验证擅长发现技术错误，但对视觉层次、留白均衡、标题与主体关联、跨页节奏和“图形是否有解释力”等软质量覆盖有限。这部分仍依赖强视觉模型或人工复核。

### 9.5 字体环境限制会削弱视觉反馈可信度

SKILL 已明确：缺少 Microsoft YaHei 时，结构与几何可以检查，但不能宣称中文视觉保真通过。即使视觉模型能力很强，若观察的是字体替换后的渲染图，反馈也可能偏离目标 Office 环境。

## 10. 降低高模型依赖风险的改进优先级

### P0：建立结构化内容模型与可验证映射

在 `source.md` 和 `layout_plan.json` 之间增加 `content_model.json`，至少表达：核心结论、证据、关系类型、重要度、受众疑问、来源引用和不可删减项。再自动检查每页是否覆盖不可删减项、结论是否有证据支撑。

收益：降低内容遗漏和大纲漂移；让弱一些的模型也能沿清晰中间状态工作。

### P0：增加版式语义适配检查

为常用结构加入可确定判断的最小契约，例如：

- `timeline` 必须有时间或阶段顺序；
- `matrix` 必须声明两条轴和象限含义；
- `swimlane` 必须声明至少两个 owner；
- `flywheel` 必须声明回到起点的强化关系；
- `comparison` 必须有共享比较维度。

收益：把“选对版式”的一部分从模型直觉转成可验证规则。

### P1：引入页面级组件参数 schema

让内置 renderer 接受受约束的页面 schema，而不是直接接收自由文本列表；限定标题长度、节点数量、每节点字数、强调项和来源区。生成前先验证 schema。

收益：减少 SVG 几何返工，稳定可读性，并保留模型在内容和结构选择上的创造空间。

### P1：建立渲染反馈的逐页差异化复核

将结构报告、SVG 元数据、layout plan 和页面 PNG 汇总为逐页 QA 输入，要求视觉模型输出“证据位置 + 问题类型 + 建议动作”，禁止只给模糊审美分数。

收益：提升视觉复核的一致性，并让返工更容易定位到内容、布局或渲染环境。

### P1：建设基准题库与成品评分样例

覆盖 briefing、instructional、narrative 三种模式，以及时间线、架构、对比、流程、指标、风险等常见内容。每个样例保存源内容、确认决策、content model、layout plan、最终 SVG、渲染图和人工评审结果。

收益：可真实比较不同模型在“理解、规划、版式选择、SVG、复核”各阶段的能力，而不是只比较最终文件是否生成。

### P2：自动化可确定的文案约束

增加标题是否为结论句、页面标题重复、来源缺失、术语不一致、数字口径不一致、正文长度分布等检查。此类检查不能代替写作判断，但能降低低级错误率。

## 11. 建议的模型分工策略

不建议让一个模型以一次长上下文完成全部高依赖任务。更稳妥的分工是：

| 角色 | 负责内容 | 必须读取的证据 |
|---|---|---|
| 内容架构角色 | 源内容、受众、取舍、逐页大纲 | `source.md`、用户确认、来源引用 |
| 版式规划角色 | 页面关系、layout 类型、节奏、多样性 | 大纲、`svg-layout-catalog.md`、内容模型 |
| SVG 创作角色 | Office-safe 几何、层级、可编辑元素 | `layout_plan.json`、视觉规则、设计锁 |
| 工程执行角色 | CLI 构建、验证、日志、恢复 | 脚本输出、JSON/Markdown 报告、状态文件 |
| 视觉评审角色 | contact sheet、可疑页、跨页一致性 | PNG、layout plan、质量报告、字体环境 |

即使由同一个模型顺序承担，也应保留这些阶段边界，避免创作者在没有独立证据的情况下自行宣布作品通过。

## 12. 最终判断

当前 SKILL 的工程工作流已经较成熟地解决了“如何稳定生成一个结构正确、模板保真、正文可编辑、可验证、可恢复的 PPTX”。在这部分，模型能力比重已经较小，核心依赖是脚本可靠性、格式兼容性、测试覆盖和渲染环境。

当前仍高度依赖模型的，是“生成什么内容、为什么这样组织、为什么使用这种视觉结构、成品是否真正清晰有说服力”。这些环节决定最终成品上限，也是不同模型之间最容易拉开差距的地方。

因此后续优化不应继续把主要精力放在让模型记住更多构建命令，而应优先：

1. 把内容理解结果结构化；
2. 把版式语义适配变成部分可验证；
3. 把自由 SVG 创作收敛为“受约束组件 + 必要的定制几何”；
4. 把渲染后的视觉判断变成有证据、可定位、可回归的反馈闭环。

这样可以同时保留强模型带来的内容与视觉上限，并继续降低成品对模型临场稳定性的依赖。
