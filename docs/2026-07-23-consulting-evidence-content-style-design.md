# 咨询证据型内容生成风格设计（参考 2025 麦肯锡 AI 调研）

> 状态：已实施最小可用版本（2026-07-23）；实现范围见第 7 节，未引入新的视觉主题或模板资产。
> 范围：为 GHB-PPT-SKILL 的 **Standard** 工作流增加一个显式选择、默认关闭的内容生成风格。
> 目标：借鉴附件的“结论先行 + 证据比较 + 业务含义”叙事方法，不复制麦肯锡品牌、模板、Logo、字体或版式资产。

## 1. 结论与边界

建议新增一个可选内容档案：`consulting-evidence-cn-v1`（面向用户可称“咨询证据型内容”）。它控制**怎样组织论点和证据**，不控制 GHB 的视觉皮肤。

选择该档案后，生成内容应做到：

1. 每页先给可证伪的业务结论，而非主题名；
2. 用一个主证据结构解释结论，并保留数据口径、时间、样本或来源；
3. 在事实、推断和建议之间明确划线；
4. 通过“现状/差距 → 分群比较 → 驱动因素 → 行动含义”建立叙事，而不是把资料逐页搬运。

它**不会**做以下事情：

- 修改 `visual_style`、GHB 模板、封面、母版、品牌色、字体或 SVG/OOXML 构建链；
- 将所有页面强行画成蓝色条形图或咨询公司报告页；
- 把来源中没有的数字、因果关系或建议伪装为事实；
- 对未选择该档案的项目改变任何默认内容风格或校验结果。

这是一种“内容方法”，不是“麦肯锡官方模板”。这样既保留附件最有价值的分析结构，也避免把 A4 研究报告的视觉样式生硬复制到 16:9 GHB 演示文稿中。

## 2. 依据：附件中可复用的内容规律

分析范围是附件的报告正文，排除 PDF 第 2 页分发广告和第 21 页封底。下表中的页码均指附件 PDF 页码，而非其页脚页码。

| 观察 | 附件证据 | 应转化成的生成规则 |
|---|---|---|
| 开篇先回答“发生了什么”，再列核心洞察 | PDF 第 3 页用“AI 已进入业务实践、但多数机构仍在规模化与企业级价值早期阶段”统领 6 条洞察 | 封面后的首张正文页应给出 1 个总判断，后续页只负责证明、拆解或转化该判断 |
| 标题是结论，不是栏目 | PDF 第 4 页的“AI 应用持续拓展，整体仍处试点阶段”；第 5 页直接说明各职能规模化均未超过 10% | 标题采用“事实/比较 + 含义”句式；禁用“AI 应用现状”“高绩效企业分析”这类无结论标题 |
| 数据必须靠比较制造信息量 | PDF 第 4 页把 88% 常态化使用与 31% 规模化、7% 全面部署并置；第 13 页给出 55% 对 20%、2.8 倍 | 每页只保留支撑结论的比较基准：时间、分群、目标值或前后阶段；不能只孤立展示一个大数字 |
| “高绩效”不是标签，而是定义、对比与机制 | PDF 第 11 页定义高绩效企业约占样本 6%；第 12–15 页依次比较转型目标、工作流重塑、领导投入、人工核验和管理实践 | 涉及领先者/标杆时，必须先给分组定义，再给差异数据，最后说明可采取的机制，不能只罗列最佳实践 |
| 叙事最终落到经营含义 | PDF 第 12–15 页把增长/创新目标、工作流重构、领导背书、人机核验与价值表现串联；第 19 页再以风险治理收束 | 每个章节至少有一页把证据翻译为“所以什么”；建议必须标注为建议，且能回指前面的证据 |
| 数据可信度被显式保留 | 各图保留样本、口径和资料来源；PDF 第 20 页说明调研时间、样本量、覆盖范围和加权方式 | 事实型页面保留最小来源行；数字不得脱离时间、分母或样本而单独引用 |

附件也说明了不应照搬的部分：它是 A4 研究报告，存在双栏正文、密集图注和完整实践清单（尤其 PDF 第 15 页）。在 16:9 汇报中，这些应被压缩为结论、主图和必要注释；需要完整清单时拆成两页或放附录，不能缩小字体硬塞。

## 3. 与当前 Skill 的契合点

当前 Standard 工作流只要求作者维护 `brief.json + deck_plan.json`，且 `deck_plan.json` 已承担故事、风格和逐页设计意图。这正是内容档案的合适入口：不新增第三份作者合同，也不修改严格模式的既有视觉合同。

当前实现中，`brief.json.visual_style` 是必填视觉描述，`deck_plan.json.style` 已含 `tone`、`density` 和 `variation`；投影器会把每页的 `message`、`source_refs`、`items` 和 `reason` 写入兼容的 `layout_plan.json`。因此不要把“咨询证据型”塞进 `visual_style`：那会把内容方法和模板外观混为一谈，也会误伤已有 `professional-modern` 等选择。

## 4. 档案选择与兼容策略

### 4.1 选择方式

仅当用户明确说“咨询型 / 麦肯锡式内容 / 结论先行并以数据论证”，或确认时主动选择时，写入：

```json
{
  "schema": "ghb.deck-plan.v1",
  "style": {
    "tone": "专业、结论先行、证据驱动",
    "density": "balanced",
    "variation": "high",
    "content_profile": "consulting-evidence-cn-v1"
  }
}
```

`brief.json.visual_style` 仍保留用户原来的视觉选择，例如 `professional-modern`。若 `content_profile` 缺失，行为就是当前行为；不得通过默认值、迁移脚本或 CLI 开关自动启用它。

在一次确认中，将现有“视觉风格”问项表述为“视觉风格 + 是否启用咨询证据型内容”。这不是增加新的强制确认闸门：用户不选即不写该字段。

### 4.2 作用范围

首版只支持 **Standard**：它已有紧凑的作者合同和确定性投影能力。Quick 可以在提示中自愿遵循，但不增加校验；Strict 不新增一套合同，后续如有需求再把同样的逐页字段映射到现有 `content_model.json` 与 `layout_plan.json`。

### 4.3 不变性承诺

| 项目 | 选择档案时 | 未选择档案时 |
|---|---|---|
| GHB 模板、主题、母版、封面与致谢页 | 沿用现有流程 | 完全不变 |
| `visual_style`、字体、颜色、安全区 | 沿用用户/模板选择 | 完全不变 |
| `brief.json` 与 `deck_plan.json` 的必填字段 | 不增加必填字段 | 完全不变 |
| 构图 | 按证据语义选择现有 layout catalog；不强制蓝色图表 | 完全按现有规则 |
| SVG、DrawingML、渲染与质量门 | 继续使用现有链路 | 完全不变 |

## 5. 逐页内容合同

选择档案后，每张正文页仍以现有 `message` 作为主结论，并可在该页追加一个小型、可选的 `consulting_content` 对象：

```json
{
  "id": "adoption-scale-gap",
  "page": 2,
  "type": "data-story",
  "message": "AI 应用已普及，但规模化仍是企业级价值的主要缺口",
  "layout": "editorial",
  "visual_priority": "88% 常态化使用与 38% 规模化或全面部署的落差",
  "source_refs": ["sources/source.md#图1-AI应用阶段"],
  "consulting_content": {
    "claim_type": "fact",
    "evidence": [
      {
        "label": "至少一个职能常态化使用 AI",
        "value": "88%",
        "source_ref": "sources/source.md#图1-AI应用阶段"
      },
      {
        "label": "规模化或全面部署",
        "value": "38%",
        "source_ref": "sources/source.md#图1-AI应用阶段"
      }
    ],
    "so_what": {
      "type": "inference",
      "text": "下一阶段的讨论重点应从单点试验转向跨工作流的规模化条件。"
    }
  }
}
```

这个对象不重复 `message`，只补足它的证据和含义。字段约束保持很小：

| 字段 | 规则 | 用途 |
|---|---|---|
| `claim_type` | `fact`、`inference`、`recommendation` 之一 | 明确事实与判断的边界 |
| `evidence` | `fact` 至少 1 条；每条含 `label`、`value`、`source_ref`；建议默认 1–3 条 | 防止无来源的大数字和“结论先行但无证明” |
| `so_what` | 可选；类型为 `inference` 或 `recommendation` | 将图表翻译为业务含义，而不把含义伪装成原始数据 |
| `source_refs` | 继续使用现有字段，至少覆盖该页事实来源 | 与当前投影、内容模型和回读链兼容 |

不为所有页面强制写 `consulting_content`：纯章节锚点、过渡页和已由 `items` 完整表达的流程页可以只保留 `message + source_refs`。但所有事实型页都必须能在页面或备注中回溯到来源。

## 6. 内容生成规则

### 6.1 标题与论点

先写结论，再选图。标题优先使用下面四种句式；必要时可两行，但仍是一句话。

| 场景 | 标题句式 | 附件对应方式 |
|---|---|---|
| 现状有落差 | `X 已经…，但 Y 仍…` | “AI 应用持续拓展，整体仍处试点阶段” |
| 分群有差异 | `A 比 B 更…，差异来自…` | 高绩效企业与其他企业的 2.8 倍工作流重塑差异 |
| 驱动因素 | `要实现 Z，关键先做 A/B/C` | 高绩效企业的工作流、领导、核验与实践机制 |
| 决策建议 | `下一步应…，以解决…` | 由前页证据推导出的行动，而非数据事实 |

禁止从章节名直接充当标题，例如“市场现状”“高绩效企业”“研究发现”。若源资料只有主题没有结论，应先把它留在 `items` 或附录，而不是编造结论。

### 6.2 证据选择与页面预算

- 一页只服务一个主结论，默认一个主图/主表/主结构；不是每页都需要图表。
- 优先保留能改变判断的基准：同比、目标差、领先者与其他、阶段漏斗、关键分组。单独展示 88% 不如展示 88% 与 38% 的部署落差。
- 主图上突出 1–3 个决定性数字；剩余数据仅在确有比较必要时保留。完整表格改为附录或拆页。
- `fact`：原样保留数值、单位、时点、分母和必要限定词（例如“受访者比例”“至少一个职能”）。
- `inference`：用“表明 / 意味着 / 提示”表达，并紧邻支撑它的事实；不把相关性写成因果。
- `recommendation`：用“建议 / 应考虑”表达，并连接前页已证实的问题或机会；没有业务上下文时只给可选动作，不声称唯一答案。
- 页面底部保留精简来源行。对外部调研，至少给出报告名、年份和图号/页码；具体脚注与口径保留在备注或附录。

### 6.3 叙事骨架

不强制固定页数，但优先采用附件验证过的顺序：

```text
总判断
  → 现状与规模化差距
  → 关键分群/场景差异
  → 价值是否兑现
  → 高绩效者或标杆机制
  → 对当前组织的行动、约束与风险
```

资料不足以支撑“高绩效者”比较时，跳过该段，不虚构标杆。资料是操作手册时，可将“价值是否兑现”替换为“问题 → 机制 → 落地步骤”。

### 6.4 与现有版式 catalog 的映射

内容档案只决定内容意图；具体 SVG 仍由现有 catalog 按语义选择。推荐映射如下。

| 证据关系 | 建议 `type` / `layout` | 不应使用 |
|---|---|---|
| 趋势 + 阶段缺口 | `data-story` / `editorial` 或 `waterfall` | 为了“咨询感”硬塞矩阵 |
| 两组或多组可比对象 | `comparison` / `matrix` 或并列条形比较 | 没有共同维度的卡片堆叠 |
| 价值驱动、行动链路 | `process` / `swimlane`、`layered_arch` | 把因果链写成并列 bullet |
| 结论、含义、章节锚点 | `summary`、`hero`、`section-anchor` / `editorial` | 装饰性大图替代可编辑内容 |
| 实践清单或完整数据 | 拆页、附录，或 `evidence` 表格 | 通过缩小字体复制 PDF 密集表 |

这与现有“内容语义优先于版式多样性”的原则一致；不引入新的图表渲染器，也不要求每页套固定咨询框架。

## 7. 最小实现方案（已实施）

### 7.1 文件与改动

| 文件 | 改动 | 原因 |
|---|---|---|
| `SKILL.md` | 增加一段“可选内容档案”，明确仅在 `content_profile` 被选择时读取下述参考 | 让生成代理有唯一入口，默认路径不变 |
| `references/content-styles/consulting-evidence-cn-v1.md` | 写入第 5–6 节的可执行规则、反例和来源要求 | 避免把长风格说明塞回主 `SKILL.md` |
| `references/workflow-modes.md` | 在 Standard 的 `deck_plan.json.style` 示例中标明可选 `content_profile` | 说明字段归属和默认关闭语义 |
| `scripts/workflow_profiles.py` | 仅在字段存在时校验白名单、将 `content_profile` 和 `consulting_content` 原样投影到生成的 `layout_plan.json` 与 `design_spec.md` | 防止内容档案在 Standard 投影后丢失，不改视觉合同 |
| `tests/test_workflow_profiles.py` | 增加选择/未选择/非法值三类回归 | 证明兼容性和 fail-fast 行为 |

不要新增 CLI 子命令、第三份作者合同、模板参数或全局配置。`content_profile` 是 `deck_plan.json` 的可选加法，使用未知字段可容忍的现有 schema 设计。

### 7.2 投影行为

当且仅当 `deck_plan.style.content_profile` 存在时：

1. `workflow_profiles.py` 校验其属于允许集合（首版仅 `consulting-evidence-cn-v1`）；
2. 在生成的 `design_spec.md` 增加 `Content profile: consulting-evidence-cn-v1`；
3. 将每页的 `consulting_content` 深拷贝到其生成的 `layout_plan.json` 行；
4. 生成器/作者据此完成内容卡、SVG 和来源行。

字段不存在时，不在生成文件中写空值或默认 profile。这样旧项目的 JSON 和生成行为保持原状，手写的 `design_spec.md` 继续受现有“不得覆盖”保护。

## 8. 验收与测试

实施完成需同时满足以下可观察条件：

| 验收项 | 验证方式 |
|---|---|
| 默认项目不受影响 | 用现有 Standard fixture 运行投影；断言未出现 `content_profile`/`consulting_content`，现有测试继续通过 |
| 显式选择可透传 | fixture 设置 `content_profile` 和一张 `consulting_content` 页；断言生成的 `design_spec.md`、`layout_plan.json` 都保留该信息 |
| 配置错误早失败 | 使用未知 profile 或 `fact` 缺少有效证据/来源时，在写任何投影文件前报清晰错误 |
| 内容质量可人工复核 | 用附件第 4 页的 88%/38% 事实做一张试验页：标题写出部署落差、图中可见比较、底部来源完整、`so_what` 标为推断 |
| 现有交付底线不退化 | 运行定向的 `pytest -q tests/test_workflow_profiles.py tests/test_skill_docs.py`，再运行仓库既有完整测试和 `git diff --check` |

不要为“标题是否有洞察”写脆弱的关键词评分器。标题句式、证据边界和来源完整性可做确定性检查；是否真正回答受众问题仍应在现有渲染后的逐页人工检查中确认。

## 9. 分两步落地

1. **先做最小可用版本**：新增 opt-in 档案、参考文档、投影透传和三类回归测试。它立即让 Skill 能稳定生成“结论—证据—含义”内容，同时不触碰模板与渲染链。
2. **再做小样验证**：基于附件 PDF 第 3–15 页做 5–7 页的内部试验 deck，重点检查总判断、部署缺口、标杆对比、工作流机制和行动页是否都有可追溯证据。只有确认该样例比默认内容更清晰，才考虑将档案扩展到 Quick 或 Strict。

这比把“麦肯锡风格”作为新的全局视觉主题更稳妥：用户获得可复用的分析叙事，现有 GHB 风格和成熟质量链完全保留。
