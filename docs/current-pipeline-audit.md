# Current Pipeline Audit

Audit date: 2026-07-10  
Starting commit: `2fb1e26`  
Goal branch: `codex/goal-optimize-ghb-ppt-skill`

This audit describes the repository before the goal's P0 implementation work.
It includes the pre-existing, uncommitted layout-diversity changes that were
present when the goal started; those changes are not treated as new goal work.

## Executive risk order

1. **P0 — master/layout relationship integrity.** The current merge script
   copies a distinct ending layout and points it at the injected master, but
   the injected master's `sldLayoutIdLst` and relationship file only register
   the cover and content layouts. This is an inconsistent OOXML graph even if
   tolerant readers happen to open a specific output.
2. **P0 — name and ID collision assumptions.** The merge script assumes the
   injected names `slideMaster2` and `theme2` are free, writes a fixed master ID
   `2147483649`, and copies template media using original basenames. Existing
   content parts or media with those names can be overwritten or duplicated.
3. **P0 — no final package validator.** The SVG exporter checks dangling
   relationships before writing its intermediate PPTX, but the GHB merge step
   has no equivalent full-package validation after it mutates the package.
4. **P0 — manual and non-idempotent orchestration.** Background removal is an
   inline regex in Markdown, cover/content output discovery is manual, and the
   five stages have no shared run manifest, log, checkpoint, or failure state.
5. **P0 — sample-specific and regex-fragile behavior.** Documentation mentions
   33 body slides; XML mutation depends on exact namespace prefixes, attribute
   ordering, and exact relationship target spelling.
6. **P1 — final-layout checks do not inspect PPTX geometry.** Existing checks
   operate on source/final SVGs. They do not measure final DrawingML objects,
   detect full-slide pictures in the merged deck, verify master mounting, or
   compare source text with PPTX text.
7. **P1 — render verification is absent.** A `soffice` command is available in
   the bundled runtime and Keynote is installed, but no repository command
   renders final decks or records which renderer was used.
8. **P2 — instruction drift.** `SKILL.md` and `README.md` duplicate long command
   sequences, refer to missing upstream references in some vendored docs, and
   describe checks that are not enforced by the GHB scripts.

## 1. Real five-stage call chain

### Phase 1 — cover

```text
template
  -> template_fill_pptx.py analyze
       -> template_fill_pptx.cli
       -> analyzer.analyze_pptx
       -> slide_library.json
  -> human/agent writes cover_fill_plan.json
  -> template_fill_pptx.py check-plan (documented only indirectly)
  -> template_fill_pptx.py apply
       -> cli._timestamped_pptx_path
       -> applier.apply_plan
       -> cloned one-slide cover_<timestamp>.pptx
  -> manual output rename/move
  -> fix_cover_font.py mutates cover PPTX in place
       -> scans slide XML only
       -> replaces Arial Unicode MS / Arial Black / 楷体 / Arial
       -> Microsoft YaHei
```

Observed template analysis: four slides and cover slots `s01_sh8` (title),
`s01_sh6` (subtitle), `s01_sh4` (date). The analyzer emits the documented slot
IDs, but `apply -o cover.pptx` always writes a timestamped sibling.

### Phase 2 — content planning and authored SVG

```text
project_manager.py init
  -> projects/<name>_ppt169_<date>/
source import (optional)
  -> project_manager.py import-sources
  -> source_to_md adapter selected by suffix/URL
content confirmation (manual blocking gate outside scripts)
  -> design_spec.md + spec_lock.md + layout_plan.json
optional assets
  -> image_search.py and/or image_gen.py and/or bundled icons
agent-authored SVG
  -> svg_output/*.svg
quality gates
  -> svg_quality_checker.py
  -> check_layout_diversity.py
  -> visual_asset_checker.py --stage authored
```

The SVGs are not generated by a deterministic repository program in the normal
workflow; an agent writes them after the content confirmation gate. The demo
projects are deterministic generators, but they are examples rather than a
general content-to-SVG implementation.

### Phase 3 — content PPTX

```text
inline regex from SKILL.md mutates svg_output/*.svg
  -> removes the first exact <g id="bg"><rect .../></g>
total_md_split.py
  -> notes/total.md -> notes/<svg-stem>.md
finalize_svg.py
  -> copies svg_output/ to svg_final/
  -> embeds icons
  -> aligns/embeds raster images
  -> flattens positional text
  -> converts rounded rects
visual_asset_checker.py --stage finalized
svg_to_pptx.py
  -> svg_to_pptx.pptx_cli
  -> native mode reads svg_output/ by default, not svg_final/
  -> drawingml_converter repeats icon expansion and positional tspan handling
  -> pptx_builder creates editable DrawingML shapes
  -> exports/<project>_<timestamp>.pptx
  -> backup/<timestamp>/svg_output/
```

This split is intentional: native export preserves editable SVG primitives and
performs required preprocessing in memory; `svg_final/` is primarily the
post-processed SVG/legacy-snapshot path. Therefore background removal must
currently mutate `svg_output/` for the native content PPTX to be transparent.

### Phase 4 — GHB master merge

```text
merge_template_master.py
  inputs: content PPTX, template PPTX, filled cover PPTX,
          content-layout index, optional ending choice
  -> loads all three ZIP packages into memory
  -> assumes content slideMaster1/theme1 and injects slideMaster2/theme2
  -> copies cover/content layouts and required media
  -> appends cloned cover slide part, then prepends its presentation sldId
  -> rewrites every original slide's layout Target to injected content layout
  -> optionally clones a template ending slide/layout and appends its sldId
  -> regex-mutates presentation.xml, presentation rels, and Content Types
  -> final.pptx
```

The actual GHB template has one master, five layouts, four slides, six PNG
media parts, and three theme parts. Its slide mapping is 1→layout1,
2/3→layout2, and 4→layout3. Layout2 references image1/image2; layout3 references
image3/image4/image5; the master references image2/image6 and theme1.

### Phase 5 — validation

```text
python-pptx open and manual page-count print
ppt_to_md.py final.pptx
  -> Markdown text/table/notes extraction + picture extraction
layout diversity check against source SVG
visual asset check against svg_final
manual XML/relationship checklist in SKILL.md
```

There is no GHB-specific final validator, JSON report, Markdown report, render
command, contact sheet, or final-PPTX object-layout analysis.

## 2. Inputs, outputs, and failure modes

| Stage | Required inputs | Outputs | Current failure behavior |
|---|---|---|---|
| Cover analyze/apply | valid PPTX template, valid fill plan | library JSON, timestamped cover PPTX | CLI returns 1 for common runtime errors; timestamp discovery is left to the caller; malformed ZIP/XML can surface raw exceptions |
| Font fix | writable cover PPTX | same path rewritten | no input existence/type guard; writes `<path>.tmp`; replaces all named fonts in every slide XML, including intentional Arial/Arial Black |
| Project/source import | project name, optional source | project tree and normalized Markdown/assets | several converters need optional libraries/tools/network; import records conversion errors as `skipped`; default may move a source located inside the repo unless `--copy` is used |
| SVG authoring | confirmed content choices and specs | `svg_output/*.svg` | human/agent step; no deterministic generic generator; confirmation can block unattended use without a fixed test configuration |
| SVG checks | SVGs, optional spec/plan/assets | console or limited JSON | checkers cover complementary subsets; warnings are not consistently gates; no combined report |
| Notes split | `notes/total.md`, numbered SVG names | per-slide notes Markdown | fails if any SVG lacks matching notes; therefore notes are effectively mandatory in the documented Phase 3 |
| Finalize | `svg_output`, icon/image assets | destructive replacement of `svg_final` | deletes existing `svg_final`; helper failures are sometimes counted/printed but do not always make the overall return false |
| SVG→PPTX | SVGs and python-pptx; optional render libs for legacy | timestamped native PPTX and backup | explicit output avoids timestamp, but documented path does not use it; native conversion raises on unsupported visual elements; legacy render can degrade to pure SVG |
| Master merge | three valid PPTX ZIPs and valid indices | final PPTX | many missing parts fail clearly through `must`; invalid indices and empty template ending lists can fail less clearly; output path is removed before the new ZIP is proven valid |
| Validation | final PPTX and source project | console/Markdown/assets | checks are separate and mostly manual; no authoritative exit code for total deck quality |

## 4. Hard-coded values and assumptions

- Template path/name: `templates/GHB_PPT_模板.pptx`.
- GHB template slide roles: cover slide 1, content layout 2, ending last slide
  (documented as slide 4/layout3).
- Cover slot IDs: `s01_sh8`, `s01_sh6`, `s01_sh4`.
- Brand: `#AB1F29`, `#44546A`; fonts Arial Black/Microsoft YaHei/Arial.
- Canvas: SVG `1280×720`; PPT uses 16:9 dimensions derived at 96 px/in.
- Chrome panel coordinates: `(56,96,1168,608)` and fixed footer/header positions.
- Background removal accepts only a narrow exact `id="bg"` group pattern.
- Merge assumes content has `slideMaster1`, `theme1`, and relative layout targets
  spelled `../slideLayouts/slideLayoutN.xml`.
- Merge injects fixed `slideMaster2`, `theme2`, and fixed master ID
  `2147483649`; it assigns new layouts from the current maximum index.
- Merge copies media by basename without collision-safe renaming.
- Template master is assumed to be `slideMaster1` and template theme `theme1`.
- The docs' “33 body slides” sentence is sample-specific; implementation uses
  the actual counted body-slide parts and is not itself fixed to 33.
- Layout diversity review signals: three identical consecutive layouts or
  fewer than four types in an 8+ slide deck are reported as non-blocking
  advice; missing `data-layout` metadata remains an error.
- SVG text density thresholds and collision overlap ratio are fixed heuristic
  constants, not PPTX layout measurements.

## 6. Implemented behavior not accurately or fully documented

- Native SVG export reads `svg_output/` by default; `finalize_svg.py` output is
  not the default native source. Preprocessing is repeated in memory for icons
  and positional tspans.
- Native export already validates internal relationship targets before writing
  the intermediate PPTX and replaces the public output only after packaging.
- Native export can take an explicit `-o`, eliminating the need to discover a
  timestamped content output; current GHB docs do not use that capability.
- The exporter writes conversion traces, backups, metadata, transitions,
  animations, notes, narration, and media deduplication, but the GHB workflow
  exposes only a subset.
- The project importer namespaces extracted source images to prevent collisions,
  but the final GHB merge does not apply equivalent media collision handling.
- The current visual checker enforces a `layout_plan.json` density contract and
  authored/finalized image/icon rules, but it does not validate the final deck.

## 9. Highest-risk OOXML operations

- Regex-based parsing/reconstruction of relationship XML; attribute order or
  namespace-prefix changes can make relationships disappear.
- Fixed master/theme names and fixed master ID allocation.
- Reusing template media basenames inside a content package.
- Copying a layout without registering it in the master's layout list/rels.
- Mutating `presentation.xml` by string replacement rather than XML nodes.
- Appending content-type overrides without duplicate/part-existence checks.
- Adding tag parts while retaining original tag filenames, which can collide.
- Dropping cover/ending notes while intermediate body notes may lack a
  notesMaster, producing reader-dependent behavior.
- Removing an existing output before the merged package has passed validation.

## 10. Current quality-check capability and blind spots

Can detect before merge:

- XML well-formedness, viewBox/dimensions, many unsupported SVG constructs,
  missing images, some low-resolution assets, unresolved icons, invalid font
  stack tails, spec-lock drift, and missing web-image attribution.
- Declared SVG QA-box collisions, simple untransformed shape overflow,
  mojibake markers, authored/finalized raster policy, layout-plan mismatch,
  text-density thresholds, and layout repetition.
- Unsupported SVG visuals and dangling intermediate-PPTX relationships during
  native export.

Cannot currently detect or prove:

- Final ZIP/relationship/content-type graph correctness after GHB merge.
- Correct cover/content/ending layout and master mounting.
- Duplicate slide IDs, relationship IDs, or parts in the final package.
- Missing final media caused by basename collisions.
- Final object counts, object bounds, overlaps, blank pages, full-slide images,
  text editability, or actual DrawingML font/color usage.
- Source-to-final text preservation, placeholders, or repeated/missing titles.
- Speaker-note support in the merged result.
- PowerPoint/LibreOffice/Keynote visual fidelity, font substitution, crop,
  occlusion, or aesthetic quality.

## Deterministic baseline design

The baseline will use repository-owned, offline fixtures and will never call
image search or image generation.

| Fixture | Body content | Primary purpose |
|---|---|---|
| A technical sharing | section, concept, process, comparison table, architecture, summary | representative enterprise technical deck |
| B management plan | current issues, goal, three initiatives, roadmap, metrics, next actions | management narrative and varied page structures |
| C layout stress | long mixed-language titles/body, complex table, many cards/nodes, source/footer text | overflow, density, small text, and extraction pressure |
| D OOXML matrix | deterministic minimal editable SVGs in counts 1/3/10; variants for no ending, explicit ending, no image, bundled icon/image | merge graph and allocation regression |

Baseline artifacts will be versioned by phase (`before/` and `after/`) so later
builds cannot overwrite the pre-change evidence. Each case will retain source,
SVG, cover/content/final PPTX, key OOXML summary, validator output, and render
artifacts when a reliable renderer succeeds.
